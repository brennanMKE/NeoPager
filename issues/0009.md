## Description

Real command output has lines wider than the terminal, and users resize windows mid-session. Both must behave deliberately instead of letting the terminal wrap arbitrarily and break the viewport math.

## Expected behavior

- Lines wider than the terminal are truncated at the right edge (phase 1 policy; soft-wrap can be a later option). Truncation must not corrupt layout or bleed escape bytes.
- On terminal resize (SIGWINCH), the viewport height and width recompute, the visible slice re-renders, and `offset` re-clamps so the view stays valid (e.g. shrinking near the bottom doesn't leave blank rows).
- The status bar stays pinned and correct through resizes.

## Notes

Truncation-not-wrap keeps the model simple: 1 buffer line == 1 screen row, so offset math and the position indicator stay exact. Wrapping would decouple buffer lines from screen rows and is a real chunk of work — if it's wanted later, file it as its own ticket then. Check whether SwiftTUI observes SIGWINCH already; tab-width handling (expand tabs to spaces on load) belongs here too since tabs break column truncation.

## Root cause

Real command output has lines wider than the terminal and users resize mid-session;
without deliberate handling the terminal wraps arbitrarily and breaks the viewport
math, and tabs throw off column truncation.

## Fix

Three parts, keeping the invariant 1 buffer line == 1 screen row:

- **Truncation:** `PagerView` cuts each line to `viewportWidth` columns (no wrap), so
  one buffer line stays one screen row and offset/position math stays exact.
- **Tabs:** `ContentSource.expandTabs` expands tabs to spaces at load time using
  8-column tab stops, so tabs never reach the renderer to break column counting.
- **Resize:** a `SIGWINCH` `DispatchSource` recomputes `PagerState` viewport
  height/width from `TIOCGWINSZ` and re-clamps the offset (SwiftTUI re-lays its own
  layer, but only `PagerState` tracks the pager's geometry/offset). Held via
  `withExtendedLifetime` for the process lifetime.

Wrapping was deliberately deferred (now filed as #0019, which makes wrap the default
and chop a toggle, with horizontal scroll in #0016).

## Verification

- Truncation: a 200-character line on an 80-column terminal renders as exactly 80
  columns, no wrap (pty harness, longest run of the marker char == 80).
- Tabs: 11 `ContentSourceTests` cover tab expansion to 8-col stops at various
  positions, plus line splitting / CRLF / lossy UTF-8 (`swift test`, green).
- Resize: a pty harness shrinks the window (TIOCSWINSZ) and sends `SIGWINCH`; the
  status bar re-renders at the new size, the pager stays responsive to further keys,
  and Esc still exits 0.

## Gotchas

Truncation is currently by `Character` count, which equals display columns only for
narrow ASCII — wide (CJK) characters and ANSI escapes need column-aware slicing
(#0012). NeoPager installs its *own* `SIGWINCH` source in addition to SwiftTUI's; both
fire and converge because both read the same `TIOCGWINSZ`. Eyes-on resize behavior on
a real terminal (no blank rows near the bottom after shrinking) is the user's check
for `closed`.

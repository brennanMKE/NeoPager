## Description

`less` wraps long lines onto continuation rows by default and chops them only with `-S`. NeoPager currently does the opposite — it chops (truncates) as its phase-1 policy (#0009). Switch the default to wrap to match `less` expectations, keeping chop available as a mode.

## Expected behavior

- By default, a buffer line wider than the terminal soft-wraps onto continuation rows; no text is hidden.
- A toggle switches to chop mode (the current truncation behavior of #0009), in which horizontal scroll (#0016) applies. Provide at least an `-S` startup flag; a runtime toggle key is a nice-to-have.
- The position indicator keeps reporting **buffer** line numbers (`line/total`), not wrapped-row numbers.

## Notes

This is the real-work item #0009 deliberately deferred: *"wrapping would decouple buffer lines from screen rows and is a real chunk of work — file it as its own ticket."* Wrapping breaks the current invariant that one buffer line == one screen row, so the scroll model (#0006) and viewport slice (#0005) must be reworked to operate on a **wrapped display-row index** rather than a buffer-line offset:

- Build a mapping between buffer lines and wrapped display rows for the current width; recompute on resize / width change (#0009).
- `offset`, `viewportHeight`, page/half-page math, and `g`/`G` (#0015) all move in display-row space; the status bar translates the top display row back to a buffer line number.
- Wrapping must count **display columns**, not bytes/characters, so it composes with ANSI color (#0012) and wide characters.

Supersedes the PRD's "truncate is the phase-1 policy" decision — update `PRD.md` (the non-goal note about wrapping) when this lands.

## Root cause

NeoPager chopped long lines (#0009's phase-1 policy) while `less` wraps by default,
so wide output hid text off the right edge unexpectedly.

## Fix

Reworked `PagerState` around a **display-row** model. `rebuildDisplayRows()` flattens
the buffer into `[DisplayRow]`, where each row carries its source `bufferLine` and its
text; a buffer line wider than `viewportWidth` is split into width-wide segments (wrap),
while chop mode (or `viewportWidth == 0`) yields exactly one row per line. The scroll
`offset`, `maxOffset`, page/half-page/jump math, and `visibleLines()` all operate in
display-row space; `viewportWidth == 0` collapses to the old 1:1 model, so existing
behavior and tests are preserved. `bottomBufferLine` maps the bottom visible row back to
a buffer line so the status bar still reports `line/total` in buffer lines.

Wrap is the default; the `-S` / `--chop-long-lines` flag selects chop. `setViewport`
(resize) and `setWrap` re-wrap for the new width/mode while anchoring the top buffer
line so the reading position stays put across a reflow.

The runtime wrap-toggle key was left out (the issue marked it a nice-to-have); only the
`-S` startup flag ships. `PRD.md` needed no change — it never documented a truncate
policy.

## Verification

`swift test` (53 total): 7 new wrap tests cover splitting a 25-char line into rows at
width 10, chop = one row per line, width-0 = no wrap, `bottomBufferLine`, scrolling
through display rows, and top-line preservation across both a wrap toggle and a resize.
End-to-end under a pty harness (80-col terminal, a line with a marker at column 150):
the marker is visible on a continuation row by default (wrapped) and truncated away
under `-S`.

## Gotchas

Wrapping currently splits by `Character` count, which equals display columns only for
narrow ASCII — wide (CJK) characters and ANSI escapes need column-aware splitting
(#0012). `DisplayRow` had to be marked `nonisolated` (like `PagerState`) so tests can
read `.bufferLine` from a nonisolated `XCTAssert` autoclosure under the package's
`MainActor` default isolation. Horizontal scroll (#0016) only applies in chop mode now
— there's nothing hidden to scroll to when wrapping.

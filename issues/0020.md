## Description

The status bar advertises the core keys, but the full set — `g`/`G`, `d`/`u`, `b`, `q`, horizontal scroll, (and search, if added) — won't fit on one row. `less` shows a help screen via `h`. Add an overlay that lists every binding, keeping the always-visible status bar uncluttered.

## Expected behavior

- `h` (and F1) opens a help overlay listing all key bindings, grouped (movement, paging, jump-to-end, quit, and search if present).
- Any key (or Esc, or `h` again) closes the overlay and returns to the pager at the same scroll position — closing help must **not** exit the pager.
- The overlay is visually distinct (e.g. a centered box or a full-screen list) and survives a terminal resize.

## Notes

`h` is a printable character → `.char("h")` in the key handler, toggling a `PagerState.showingHelp` flag (or a small view-mode enum). F1 is an escape sequence (`ESC O P` or `ESC[11~`) and needs a `RawKeyParser` addition.

Render via a SwiftTUI overlay (e.g. a `ZStack` over `PagerView`, or swapping the root content) driven by the flag. Use `h` rather than `?`, which `less` reserves for backward search (#0010).

Keep the listed bindings in sync as keys are added (#0013, #0014, #0015, #0016, #0017, and #0010 if search lands). Phase-2 caveat: in search-input mode, `h` is a literal query character.

## Fix

`PagerState.showingHelp` (toggled via `setShowingHelp`, scroll position untouched) plus
a `HelpView` listing every binding (movement, paging, half-page, jump, horizontal
scroll, help, quit) as a padded two-column list. `PagerView.body` swaps to `HelpView`
when `showingHelp` is true. `h` and F1 open it; while showing, the key handler is modal
— it intercepts at the top and closes help on **any** key (so Esc/q close help rather
than quitting). `RawKeyParser` gained `.f1` (SS3 `ESC O P` and CSI `ESC[11~`). `h`
(not `?`, which `less` reserves for backward search) is the primary key.

## Verification

`swift test` (59 total): a `PagerState` test asserts the help flag toggles while the
offset is preserved; a `RawKeyParser` test asserts F1 decodes in both forms. End-to-end
under a pty harness: pressing `h` shows the overlay (header "key bindings" + "Press any
key to close"); pressing a key returns to the pager with the `Esc quit` status bar back.

## Gotchas

The overlay is a static full-screen list (any key closes), not a scrollable sub-pager
like `less`'s help — adequate for a short binding list. When search lands (#0010), `h`
must become a literal query character in search-input mode, and the help list should
gain the search keys. Keep the `HelpView.bindings` list in sync as keys are added.

# 0020 — Help overlay: h and F1

## Description

The status bar advertises the core keys, but the full set — `g`/`G`, `d`/`u`, `b`, `q`, horizontal scroll, (and search, if added) — won't fit on one row. `less` shows a help screen via `h`. Add an overlay that lists every binding, keeping the always-visible status bar uncluttered.

## Expected behavior

- `h` (and F1) opens a help overlay listing all key bindings, grouped (movement, paging, jump-to-end, quit, and search if present).
- Any key (or Esc, or `h` again) closes the overlay and returns to the pager at the same scroll position — closing help must **not** exit the pager.
- The overlay is visually distinct (e.g. a centered box or a full-screen list) and survives a terminal resize.

## Notes

`h` is a printable character → `.char("h")` in the key handler, toggling a `PagerState.showingHelp` flag (or a small view-mode enum). F1 is an escape sequence (`ESC O P` or `ESC[11~`) and needs a `RawKeyParser` addition.

Render via a SwiftTUI overlay (e.g. a `ZStack` over `PagerView`, or swapping the root content) driven by the flag. Use `h` rather than `?`, which `less` reserves for backward search (#0010).

Keep the listed bindings in sync as keys are added (#0013, #0014, #0015, #0016, #0017, and #0010 if search lands). Phase-2 caveat: in search-input mode, `h` is a literal query character.

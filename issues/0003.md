## Description

In `cmd | neopager`, fd 0 is the pipe, not the keyboard — so after the piped content is read to EOF, SwiftTUI would be reading key events from an exhausted pipe and the pager would be uncontrollable. Interactive input must be rebound to the controlling terminal. This is the project's #1 technical risk; treat it as a spike and prove it before building anything on top.

## Expected behavior

After `ContentSource` drains the pipe, the pager opens `/dev/tty` and makes it the source of keyboard input (e.g. `dup2` onto fd 0, or pointing SwiftTUI's input handling at the tty fd) before `Application(rootView:).start()` runs. `git log | neopager` then responds to keystrokes exactly as `neopager file.txt` does.

## Notes

This is what `less` does. The open question is how SwiftTUI acquires its input file descriptor and terminal (raw mode via `tcsetattr`, etc.) — read SwiftTUI's `Application`/run-loop source in `.build/checkouts/SwiftTUI` to find where it reads stdin. If it hardcodes fd 0, `dup2(open("/dev/tty"), 0)` before `start()` is likely sufficient and requires no fork. If it uses `FileHandle.standardInput` in a way that caches state earlier, a patched fork may be needed — decide here, since [#0004](0004.md) builds on the same input layer. Verify both entry modes manually.

## Root cause

The #1 technical risk: SwiftTUI reads keyboard input from `STDIN_FILENO` (fd 0)
via a `DispatchSourceRead` created inside `Application.start()`, reading with
`FileHandle.standardInput.availableData`. In `cmd | neopager`, fd 0 is the pipe,
which #0002 drains to EOF — so without intervention SwiftTUI reads from an
exhausted pipe and the pager is uncontrollable.

## Fix

`TTYInput.reattachToControllingTerminal()`: when `isatty(STDIN_FILENO) == 0`, open
`/dev/tty` and `dup2` it onto fd 0, then close the extra fd. Called in `run()`
*after* `ContentSource.load` drains the pipe and *before* `Application.start()`.
Because SwiftTUI creates its input `DispatchSource` on fd 0 inside `start()` (after
the redirect) and reads fd 0 live, no fork is needed — this is exactly what `less`
does. No-op when stdin is already a TTY (`neopager file.txt`).

## Verification

The mechanism was proven by a standalone controlling-tty spike before any UI was
built: a probe drained a piped fd 0 to EOF, did `open("/dev/tty") + dup2`, then read
a key via `FileHandle.standardInput.availableData` (SwiftTUI's exact call) — it
returned the injected keystroke (`GOT:"X\n"`), disproving the theory that
`FileHandle` caches fd 0. End-to-end, the full pager under a controlling-pty
harness (`cmd | neopager`, stdin a pipe) renders and responds to injected
keystrokes (Down/PageDown/Option-Down/Esc all decoded; `q`/Space/`b` paging; Esc
exits 0) — i.e. the piped case behaves like the file case.

## Files changed

- `Sources/NeoPager/TTYInput.swift` — `reattachToControllingTerminal()` (the
  `dup2(/dev/tty, 0)` redirect) and `TerminalSize.current()` (TIOCGWINSZ).
- `Sources/NeoPager/NeoPager.swift` — calls the reattach before `Application.start()`.

## Gotchas

The redirect must happen after the pipe is fully drained (#0002 reads to EOF) and
before `start()`. `FileHandle.standardInput` is *not* an obstacle — it wraps
descriptor 0 and reads it live, so `dup2` onto fd 0 transparently redirects it.
Full interactive confirmation across Terminal.app/iTerm2/Ghostty is the user's
eyes-on check to move this to `closed`.

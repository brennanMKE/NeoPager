## Description

When long lines are chopped (truncated at the right edge) rather than wrapped, the text past the right edge is unreachable. `less` scrolls horizontally with `←`/`→` in its chop mode (`-S`). NeoPager truncates long lines (#0009) and already decodes `.left`/`.right` arrow events (currently no-ops in the key handler) — wire them to a horizontal offset so the cut-off text can be revealed.

## Expected behavior

- `→` scrolls the viewport right by a fixed step (e.g. 8 columns), revealing text beyond the right edge; `←` scrolls left.
- Horizontal offset clamps to `0 ... max(0, widestVisibleLine - viewportWidth)`; movement past either end is a no-op.
- All visible rows shift together; vertical position and the status bar are unaffected.
- Only meaningful while lines are chopped. When wrap mode (#0019) is active, horizontal scroll is disabled / a no-op (nothing is hidden to scroll to).

## Notes

Add a `horizontalOffset` to `PagerState` plus a clamp bound. The cleanest bound is the width of the widest line currently in the viewport (recomputed as the vertical offset changes); a simpler global "widest line in buffer" is also acceptable. `PagerView` then slices each line `[horizontalOffset ..< horizontalOffset + viewportWidth]` instead of `prefix(width)`.

`less` scrolls by half the screen width with `←`/`→` and accepts a count; keep it simple here — a fixed column step, no count language. ANSI-aware slicing composes with #0012 (count display columns, not bytes).

## Fix

`PagerState` gained `horizontalOffset` with `scrollLeft`/`scrollRight(by: 8)`, clamped
to `maxHorizontalOffset` = `max(0, widestVisibleLine - viewportWidth)`. The horizontal
windowing moved into the model: `visibleLines()` now returns each row already sliced to
`[horizontalOffset, +viewportWidth)` in chop mode (wrap segments are width-wide by
construction), so `PagerView` renders the strings directly and no longer truncates. The
`←`/`→` arrows (previously decoded but no-ops) call `scrollLeft`/`scrollRight`.

Horizontal scroll is meaningful only in chop mode: in wrap mode `maxHorizontalOffset`
is 0 so the arrows are no-ops, and switching to wrap (or resizing) resets
`horizontalOffset` to 0.

## Verification

`swift test` (57 total): 4 tests cover windowing/reveal (`abcdefghij` at width 4 →
`abcd`→`efgh`), clamping at both ends, the wrap-mode no-op, and reset-on-wrap-toggle.
End-to-end under a pty harness in `-S` mode: a line with a marker at column ~145 is
hidden initially (80-col terminal) and becomes visible after pressing `→` twelve times
(clamped to the line's end).

## Gotchas

The clamp uses the widest *currently visible* line, so after scrolling right then
scrolling vertically onto shorter lines, those render blank past their end (matches
`less`). A fixed 8-column step (no count language) keeps it simple. Column counting is
by `Character` — wide chars / ANSI escapes need the column-aware treatment in #0012.

## Description

`less` with `-F` (commonly set via `LESS=-FRX`) doesn't page at all when the content fits a single screen — it prints the content and exits. This is distinct from NeoPager's "never auto-exit at the bottom" rule (#0006): that's about not exiting *while* paging; this is about not *entering* the full-screen pager when there's nothing to scroll. It avoids a jarring alt-screen takeover for `git status | neopager` or a five-line file.

## Expected behavior

- When the loaded content fits the screen, print it to stdout normally and exit 0 **without** entering the TUI / alternate screen.
  - In chop mode: fits when `lines.count <= terminalRows`.
  - In wrap mode (#0019): fits when the total wrapped row count `<= terminalRows`.
- Otherwise, page as usual.
- This is the default behavior. (Optional: a flag to force paging even for short content.)

## Notes

Decide this in `run()` after loading content and reading the terminal size, before `TTYInput.reattachToControllingTerminal()` and `Application.start()`. When the fits-on-screen test passes, fall back to the existing line-printing path (the same `for line in lines { print(line) }` the pre-TUI build used) and return.

Because we're not paging, the threshold uses the full terminal row count — no row is reserved for a status bar (there is no status bar in this path).

This changes #0005's "content shorter than the viewport renders fully with the status bar pinned": short content now won't enter the pager at all, so that sub-case becomes unreachable in normal use (still correct for content that is taller than the screen by only a little). Note the interaction when resolving.

Edge cases: if stdout is not a TTY, neopager isn't interactive regardless — keep current behavior. Consider whether color reset / trailing newline handling matches the paged path.

## Fix

In `run()`, after building `PagerState` (which wraps the content for the terminal
width) and reading the terminal size, a guard checks `state.rowCount <= size.rows`. If
it holds, the buffer lines are printed to stdout and `run()` returns — no
`TTYInput.reattach`, no `Application.start()`, no alternate screen. Otherwise it pages
as before. `rowCount` is wrap/chop-aware (#0019), so the test naturally accounts for
long lines: in wrap mode a wide line counts as the rows it occupies; in chop mode it
counts as one. The full terminal height is the threshold (no status-bar row is reserved
when not paging). This is the default behavior; there is no opt-out flag yet.

## Verification

Under a pty harness (24-row terminal): 5 lines and exactly 24 lines both print and exit
with **no keypress**, **no alternate screen** (`ESC[?1049h` absent), and **no status
bar**; 500 lines enters the pager (alt-screen + `Esc quit` legend present) and requires
`q` to exit. The boundary is inclusive (24 == rows fits).

## Gotchas

When stdout is not a TTY, `TerminalSize.current()` falls back to 24×80, so very short
content still prints-and-exits while taller content would (as before) try to render the
TUI to a non-terminal — that pre-existing non-TTY case is out of scope here. In chop
mode (`-S`), a long line counts as one display row for the fit test, but if it's printed
in this path the terminal itself wraps it — so a "fits" decision in chop mode can use
slightly more physical rows than counted. Matches `less -F` closely enough; revisit if
it bites. This supersedes #0005's "short content renders with the status bar pinned"
sub-case — short content no longer enters the pager at all.

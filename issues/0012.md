## Description

Phase 2: commands like `git log --color=always` or `ls -G` emit ANSI SGR color sequences. A pager that strips or mangles them (the `less` without `-R` experience) loses half the value of the output. NeoPager should render colored input correctly — or at minimum strip sequences cleanly so they never corrupt the layout.

## Expected behavior

- SGR sequences (`ESC[…m`) in piped content render as their colors/styles in the viewport.
- Color state carries across line boundaries correctly even though the viewport renders arbitrary slices (a color opened on line 100 still applies when the viewport starts at line 150).
- Truncation of long lines ([#0009](0009.md)) counts display columns, not bytes — escape sequences are zero-width — and resets style at the cut.
- Non-SGR control sequences (cursor movement, etc.) are stripped, never executed.
- Search highlighting ([#0010](0010.md)) composes with content colors without leaving stuck attributes.

## Notes

This likely needs parsing SGR runs into per-line styled spans at load time (so slicing and width math work on clean text), then mapping spans onto SwiftTUI's `Text` styling — check how much of the ANSI palette SwiftTUI's foreground/background colors can express. Decoupling this from phase 1 is deliberate: until then, content is treated as plain text, and most tools don't emit color when stdout is a pipe anyway.

## Fix

`AnsiParser` (Sources/NeoPager/Ansi.swift) parses raw lines into plain text plus
per-line `[StyleRun]` (column ranges + `Style`), threading the SGR state across line
boundaries (a color opened on line 100 still applies on line 150). Escapes are removed
from the text — so the display-row wrap/chop/window column math from #0009/#0019 is
unchanged — and tabs are expanded *inside* the parser (after escapes are stripped) so
columns stay aligned with styles. Supported SGR: 16 colors + bright (30–37/90–97,
40–47/100–107), 256-indexed (`38;5;n`), true color (`38;2;r;g;b`), bold/underline/
inverse and their resets, and `0`/empty reset. Non-SGR control sequences (cursor moves,
erases, DSR, …) are stripped and never executed.

`ContentSource.load` now returns `LoadedContent` (raw lines for the no-pager print path
#0018; plain lines + style runs for the pager). `PagerState` carries the runs
(`styleRuns(onBufferLine:)`), independent of wrapping since they live in buffer-line
column space. `PagerView` merges content styles and search highlights per column into
styled spans, maps `AnsiColor` → SwiftTUI colors (inverse swaps fg/bg; the search
highlight overrides the background so matches stay visible over colored text), and
applies bold/underline via small `boldIf`/`underlineIf` view helpers.

## Verification

`swift test` (73 total): 11 `AnsiParser` tests — escape stripping (zero-width),
color/bold/underline/inverse runs, reset-ends-run, color carried across lines, non-SGR
stripped, 256 + true color, bright, empty-params reset. End-to-end under a pty harness:
a line with an embedded non-SGR `ESC[2K` renders as contiguous plain text
("GOLDMARKTAILMARK"), the `2K` never appears (stripped), and the renderer emits SGR
color sequences.

## Gotchas

Style runs are kept in buffer-line/column space, so they survive wrap/chop/resize and
compose with search highlights via a per-column merge in the view. Tab expansion *had*
to move into the parser — expanding tabs before stripping escapes would miscount
columns. Width math counts `Character`s, so wide (CJK) glyphs still count as one column
(a known limitation, shared with #0009/#0016). SwiftTUI has no public inverse modifier,
so inverse is emulated by swapping fg/bg (defaulting to white-on-black). The actual
on-screen colors are an eyes-on check for `closed`.

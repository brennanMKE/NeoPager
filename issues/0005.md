# 0005 — Render a viewport slice of the content sized to the terminal

| | |
|---|---|
| **Status** | open |
| **Module** | Views / Pager |
| **Platform** | macOS |
| **First seen** | 2026-06-10 |

## Description

The pager view must render exactly the lines that fit: terminal height minus one row reserved for the status bar, starting at `PagerState.offset`. Rendering the whole buffer and letting the terminal clip it is not acceptable — content can be hundreds of thousands of lines.

## Expected behavior

- A SwiftTUI `PagerView` renders `lines[offset ..< offset + viewportHeight]`, one `Text` row per line, top-aligned.
- `viewportHeight` is derived from the terminal size (SwiftTUI's layout proposal or `ioctl(TIOCGWINSZ)`), minus the status bar row.
- Content shorter than the viewport renders fully with the status bar still pinned to the bottom row.
- Scrolling by changing `offset` re-renders without flicker or visible tearing.

## Notes

Keep the view a dumb function of `PagerState` (offset + visible slice) so the scroll model in [#0006](0006.md) is testable without a terminal. Check how SwiftTUI diffs/redraws rows when the slice shifts by one — if per-line `Text` identity causes full-screen repaints, an id-stable structure may be needed for performance on large files.

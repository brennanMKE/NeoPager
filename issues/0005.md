## Description

The pager view must render exactly the lines that fit: terminal height minus one row reserved for the status bar, starting at `PagerState.offset`. Rendering the whole buffer and letting the terminal clip it is not acceptable — content can be hundreds of thousands of lines.

## Expected behavior

- A SwiftTUI `PagerView` renders `lines[offset ..< offset + viewportHeight]`, one `Text` row per line, top-aligned.
- `viewportHeight` is derived from the terminal size (SwiftTUI's layout proposal or `ioctl(TIOCGWINSZ)`), minus the status bar row.
- Content shorter than the viewport renders fully with the status bar still pinned to the bottom row.
- Scrolling by changing `offset` re-renders without flicker or visible tearing.

## Notes

Keep the view a dumb function of `PagerState` (offset + visible slice) so the scroll model in [#0006](0006.md) is testable without a terminal. Check how SwiftTUI diffs/redraws rows when the slice shifts by one — if per-line `Text` identity causes full-screen repaints, an id-stable structure may be needed for performance on large files.

## Relation

## Root cause

Greenfield: no view existed to render content.

## Fix

`PagerView` renders only `PagerState.visibleLines()` — the slice
`lines[offset ..< offset + viewportHeight]` — one `Text` per row, never the whole
buffer. A `Spacer` fills any unused height below short content so the status bar is
pinned to the bottom row even when the content doesn't fill the screen. Each row's id
is its position within the viewport (0-based), so as the slice scrolls SwiftTUI
updates each row's text in place rather than treating scrolled lines as
inserts/removals. `viewportHeight` is derived from the terminal size (`TIOCGWINSZ` via
`TerminalSize.current()`, minus one row for the status bar). Long lines are truncated
to width (refined in #0009).

## Verification

Under a controlling-pty harness (40×120), the initial render (5133 bytes) shows the
content plus the pinned status bar (`Esc quit` legend + `/200` total). Scrolling
produces small delta frames (~350 bytes) rather than full-screen repaints —
confirming SwiftTUI diffs and rewrites only changed cells, so per-line `Text` identity
is *not* causing full repaints (the performance concern in Notes). Short content +
status-bar-pinned and near-bottom slices are covered by `PagerState.visibleLines()`
unit tests (#0006).

## Gotchas

The `Spacer` (not blank filler rows) is what pins the status bar for short content —
rendering empty `Text("")` rows risks zero-height collapse. Delta rendering also means
end-to-end tests can't grep for contiguous line content after a scroll (only changed
cells are rewritten); assert against the status-bar percentage instead. Visual
no-flicker confirmation on a real terminal is the user's eyes-on check for `closed`.

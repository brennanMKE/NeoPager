## Description

The pager needs its content. The two entry modes are `long-command | neopager` (read everything from the stdin pipe) and `neopager file.txt` (read the file). Today the executable just prints "Hello, world!".

## Expected behavior

- A `ContentSource` type reads all input into an array of lines before the TUI starts.
- With a file argument, the file is read; a missing/unreadable file prints a clear error to stderr and exits non-zero.
- With no argument and stdin not a TTY (`isatty(0) == 0`), the pipe is read to EOF.
- With no argument and stdin a TTY, print usage and exit — don't sit waiting on an empty terminal like `more` does.
- Input that isn't valid UTF-8 is handled lossily (replacement characters) rather than crashing.

## Notes

Phase 1 reads everything up front (like `more`; `less` streams). Streaming/incremental reading is a possible later enhancement but is deliberately out of scope — note it in the PRD rather than filing a ticket until the core pager works. Reading to EOF first also simplifies [#0003](0003.md), since the pipe is fully drained before input switches to /dev/tty.

## Root cause

Greenfield: the executable only printed "Hello, world!" and had no way to ingest content.

## Fix

Added `ContentSource` (Sources/NeoPager/ContentSource.swift): a `load(path:)` entry that reads a file when a path is given, otherwise reads the stdin pipe — but only when `isatty(STDIN_FILENO) == 0`. A TTY with no file throws `ContentError.noInput`; an unreadable file throws `ContentError.fileNotReadable`. Bytes are decoded with `String(decoding:as: UTF8.self)` (lossy U+FFFD) and split on `\n`, dropping a single trailing-newline empty line and stripping CRLF `\r`. `NeoPager.run()` maps `noInput` to an ArgumentParser `ValidationError` (prints usage, exit 64) and `fileNotReadable` to a clean stderr line + `ExitCode.failure` (exit 1). Until the viewport (#0005) lands, `run()` prints the loaded lines.

## Verification

Built with `./build.sh build`, then exercised `.build/debug/neopager`:
- Pipe: `printf 'alpha\nbravo\ncharlie\n' | neopager` → three lines, exit 0.
- File: `neopager /tmp/np_test.txt` → file contents, exit 0.
- Missing file: `neopager /tmp/does_not_exist` → `neopager: …: …no such file.` on stderr, exit 1.
- Real TTY, no arg (`script -q /dev/null neopager`): printed the usage message, exit 64.
- Invalid UTF-8 (`printf 'good\n\xff\xfe\nafter\n' | neopager`): rendered U+FFFD replacement chars, no crash, exit 0.

## Files changed

- `Sources/NeoPager/ContentSource.swift` — new `ContentSource` loader + `ContentError`; line splitting and lossy UTF-8 decode.
- `Sources/NeoPager/NeoPager.swift` — added optional `file` argument and `commandName`/abstract; wired error mapping; temporary `cat`-style output.

## Gotchas

`defaultIsolation(MainActor.self)` makes new types MainActor-isolated by default, but `ParsableCommand.run()` is nonisolated — calling `ContentSource.load` from `run()` failed to compile until `ContentSource`'s methods were marked `nonisolated`. Pure-I/O helpers in this package should be `nonisolated` so they're callable from `run()` without `MainActor.assumeIsolated`.

## Description

Esc is the single exit gesture. Quitting must restore the terminal to exactly its prior state ŌĆö raw mode off, alternate screen buffer discarded so the user's scrollback is clean, cursor visible ŌĆö and exit 0.

## Expected behavior

- A bare Esc keypress (as decoded by [#0004](0004.md)) terminates the pager from any position in the content.
- The terminal is restored: cooked mode, cursor shown, alternate screen exited (the paged content does not pollute scrollback).
- Ctrl-C behaves the same as Esc (restore, then exit) ŌĆö a pager must never leave the terminal in raw mode on interrupt.
- Exit code 0 on Esc/Ctrl-C.

## Notes

Check what SwiftTUI's `Application` already handles for teardown (it installs the TUI; does it restore on SIGINT?). `Application.start()` calls `dispatchMain()` and never returns, so exiting likely means SwiftTUI's own exit path or `exit(0)` after explicit terminal restoration. Install a SIGINT handler that performs the same restore. Verify by running `git log | neopager`, exiting, and confirming `tput smso`/typed input behave normally and scrollback is intact.

## Root cause

Greenfield: the pager had no exit gesture or teardown until the input layer (#0004)
and run loop were in place.

## Fix

The pager's `keyHandler` maps the decoded `.escape` event to `Application.quit()`
(in `NeoPager.run()`, commit f829be5). `quit()` is a small patch to the vendored
SwiftTUI (commit ed58da4) that exposes the previously-private `stop()`, which:

1. `renderer.stop()` ŌĆö writes `ESC[?1049l` (exit the alternate screen buffer, so
   paged content never lands in scrollback) and `ESC[?25h` (show the cursor);
2. `resetInputMode()` ŌĆö `tcsetattr` restoring `ECHO | ICANON` (cooked mode);
3. `exit(0)`.

Ctrl-C needs no extra code: SwiftTUI's `setInputMode()` leaves `ISIG` enabled, so
Ctrl-C still raises `SIGINT`, and SwiftTUI's `SIGINT` `DispatchSource` calls the
same `stop()` ŌĆö identical restore-then-exit-0 path as Esc.

## Verification

Built green (`swift build`). Exercised the real binary end-to-end under a
controlling-pty harness (`/tmp/exit0008.py`) that pipes 300 lines in, scrolls into
the content, then sends the quit gesture and inspects the child's exit status and
output bytes:

- **Esc** (`0x1b`): exit code **0**; output contains `ESC[?1049l` (alt-screen exit)
  and `ESC[?25h` (show cursor).
- **Ctrl-C** (`0x03`): exit code **0**; same restore sequences emitted.

`resetInputMode()`'s cooked-mode restore is a `tcsetattr` call (not observable in
the output stream) on the same code path. A human eyes-on pass (scrollback clean,
`tput`/typing normal after exit) is all that remains to move this to `closed`.

## Gotchas

`stop()`'s teardown writes the restore sequences to stdout. If the terminal isn't
draining stdout at exit (e.g. a test harness that stops reading the pty), those
writes block on a full buffer and `exit(0)` is never reached ŌĆö the process appears
to hang on quit. Real terminals drain continuously, so this only bites test
harnesses: keep reading the pty through shutdown. (This cost real debugging time
during verification.)

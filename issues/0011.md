## Description

Phase 2: once a search is active, `n` jumps to the next match and `N` (Shift-n) to the previous one, with the status bar showing the current position in the match list.

## Expected behavior

- `n` / `N` scroll the viewport so the next/previous match is visible, with the current match visually distinguished from other highlighted matches.
- The status bar shows `match 3/17` alongside the normal legend, updating as the user navigates.
- At the ends of the match list, `n`/`N` clamp (no wrap, consistent with the pager's no-wrap scroll behavior); the status bar can flash `last match`/`first match`.
- Plain scrolling (arrows/pages) keeps highlights and the active query; the next `n` continues from the current viewport position.

- Blocked by: [#0010](0010.md)

## Fix

Implemented together with search (#0010). `n` → `PagerState.nextMatch()` advances
`currentMatchIndex` and scrolls so that match's display row is at the top; `N` →
`previousMatch()` steps back. Both clamp with no wrap: at the end `nextMatch` sets
`searchStatus = "last match"` (and `previousMatch` `"first match"`) without moving.
The first `n` after a search continues from the match nearest the viewport (set by
`executeSearch`); after that it steps by index. The current match is highlighted
distinctly (yellow on black vs. white on grey for the others) via `PagerState.currentMatch`
and the span renderer. The status bar shows `match N/M` (`currentMatchIndex + 1` /
`matches.count`) alongside the legend, updating as you navigate.

## Verification

`swift test` (67 total): `nextAndPreviousMatchNavigateAndClamp` walks three matches
0→1→2, asserts `n` at the end stays on the last match and sets `last match`, and `N`
steps back. End-to-end under a pty harness: `n` advances `match 1/3` → `3/3` → flashes
`last match`, with the viewport scrolling to each match.

## Gotchas

Index-based navigation (rather than pure viewport-relative) was chosen so the end-of-list
`last match`/`first match` clamp is reliable: when the last match sits in the
clamped-bottom region, a viewport-position scheme keeps re-finding it and never reports
`last match`. The trade-off is that after a *manual* scroll, `n` continues from the last
selected match rather than the viewport — acceptable, and the first `n` after a search
still starts from the viewport.

# 0004 — Decode raw key events: arrows, Option-arrows, and Esc

|---|---|

## Description

The pager's whole interaction model is raw keys: ↑/↓ scroll one line, ⌥↑/⌥↓ scroll a page, Esc exits. SwiftTUI's input model is focus/Button-oriented (TerminalDashboard only ever uses Tab/Enter and Buttons), so a key-event path that delivers these keys to the app is needed.

## Expected behavior

A `KeyReader` (or SwiftTUI extension) decodes the byte stream from the tty into discrete events: `.up`, `.down`, `.pageUp`, `.pageDown`, `.escape`, and (for phase 2) printable characters. The pager view receives these events and updates `PagerState`.

Sequences to accept:

- Arrow up/down: `ESC [ A` / `ESC [ B` (and application-mode `ESC O A` / `ESC O B`)
- Option-arrow up/down: `ESC [ 1 ; 3 A` / `ESC [ 1 ; 3 B` (xterm modifier form), plus the `ESC ESC [ A` / `ESC ESC [ B` variant some terminal configurations send
- Bare Esc: a lone `ESC` byte

## Notes

The hard part is that bare Esc is a prefix of every other sequence. Use a short read timeout (~25–50 ms) after an `ESC` byte: more bytes → escape sequence; silence → bare Esc. Test in Terminal.app, iTerm2 (both "Option as Meta" settings), and Ghostty. How this integrates with SwiftTUI's own input loop depends on the outcome of [#0003](0003.md) — it may mean intercepting bytes before SwiftTUI's focus handling, or extending a fork. Also map PageUp/PageDown keys (`ESC [ 5 ~` / `ESC [ 6 ~`) to page movement for external keyboards — same action as Option-arrows.

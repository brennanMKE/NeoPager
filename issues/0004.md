## Description

The pager's whole interaction model is raw keys: ↑/↓ scroll one line, ⌥↑/⌥↓ scroll a page, Esc exits. SwiftTUI's input model is focus/Button-oriented (TerminalDashboard only ever uses Tab/Enter and Buttons), so a key-event path that delivers these keys to the app is needed.

## Expected behavior

A `KeyReader` (or SwiftTUI extension) decodes the byte stream from the tty into discrete events: `.up`, `.down`, `.pageUp`, `.pageDown`, `.escape`, and (for phase 2) printable characters. The pager view receives these events and updates `PagerState`.

Sequences to accept:

- Arrow up/down: `ESC [ A` / `ESC [ B` (and application-mode `ESC O A` / `ESC O B`)
- Option-arrow up/down: `ESC [ 1 ; 3 A` / `ESC [ 1 ; 3 B` (xterm modifier form), plus the `ESC ESC [ A` / `ESC ESC [ B` variant some terminal configurations send
- Bare Esc: a lone `ESC` byte

## Notes

The hard part is that bare Esc is a prefix of every other sequence. Use a short read timeout (~25–50 ms) after an `ESC` byte: more bytes → escape sequence; silence → bare Esc. Test in Terminal.app, iTerm2 (both "Option as Meta" settings), and Ghostty. How this integrates with SwiftTUI's own input loop depends on the outcome of [#0003](0003.md) — it may mean intercepting bytes before SwiftTUI's focus handling, or extending a fork. Also map PageUp/PageDown keys (`ESC [ 5 ~` / `ESC [ 6 ~`) to page movement for external keyboards — same action as Option-arrows.

## Root cause

SwiftTUI's input model is focus/Button-oriented: `Application.handleInput` hardcodes
arrow keys to move focus between controls and never delivers them to views, swallows
bare Esc, and doesn't parse Option-arrows or Page keys. There is no key-event hook,
and the render primitives are `internal`, so a downstream module can't drive its own
loop either — the input layer had to be patched (vendored SwiftTUI, per #0004's
vendoring decision).

## Fix

Added `RawKey.swift` to the vendored SwiftTUI: a `RawKeyEvent` enum and a
`RawKeyParser` that decodes an input chunk into events — plain arrows (CSI `ESC[A/B/C/D`
and SS3 `ESC O A/B/C/D`), Option/Meta-arrows (`ESC[1;3A/B` and the `ESC ESC [A/B`
variant) → page movement, Page Up/Down (`ESC[5~`/`ESC[6~`), bare Esc, Enter,
Backspace, and printable characters. Patched `Application` with
`public var keyHandler: ((RawKeyEvent) -> Void)?`; when set, `handleInput` routes the
whole chunk through `RawKeyParser` to that handler and bypasses focus handling.

Bare-Esc vs escape-sequence ambiguity is resolved by **chunk decoding** rather than a
read timeout: terminals deliver a full sequence in one `availableData` chunk, and a
lone Esc arrives alone. (The issue suggested a 25–50 ms timeout; chunk decoding is
simpler and handles the common terminal behavior — noted as a known limitation for the
rare fast-Meta-combo case.)

## Verification

13 `RawKeyParser` unit tests (`swift test`, all green) assert the decoding of CSI/SS3
arrows, both Option-arrow forms, Page keys, bare Esc, Enter/Backspace, printable chars,
mixed chunks, and Ctrl-arrow falling back to plain line movement. End-to-end, a debug
trace of the running pager confirmed `ESC[B`→down, `ESC[6~`→pageDown, `ESC ESC [B`
(Option-Down)→pageDown, and lone `ESC`→escape all reach the handler over the
`/dev/tty` path (#0003).

## Gotchas

The timeout approach the issue describes is unnecessary with chunk decoding, but the
tradeoff is real: a lone Esc immediately followed in the *same* chunk by another key
(a fast Meta combo / paste) is read as Esc. Acceptable for a pager where Esc is the
deliberate quit gesture. Cross-terminal confirmation of the physical Option-arrow
encodings (Terminal.app/iTerm2 "Option as Meta" on/off, Ghostty) is the user's
eyes-on check to move this to `closed`.
